# Expense Tracker V19

## Improvements & Bug Fixes

### 1. Persistent UI State Mechanism
- **The Issue:** Whenever the user saved a Salary or deleted an item, the script would wipe the DOM and completely rebuild it to recalculate the remaining balance and month data. This natively caused all dropdown sections to close abruptly, confusing the user and breaking the experience flow.
- **The Fix:** Implemented a persistent state controller using a global `Set()` (`openSectionIds`). Every time a user clicks to expand or collapse a month/section, the script records its unique ID into this Set. 
- **The Result:** Now, when `renderExpenses()` rebuilds the layout after a Salary is saved or an item is deleted, it checks the persistent Set and automatically re-opens any dropdowns the user previously had open. 

### 2. "No Expenses" Empty State Handling
- **The Issue:** Deleting the very last expense inside an open dropdown would cause the dropdown to close (due to the above issue). Additionally, deleting the *absolute last expense* of a month completely deleted the month container, even if the user still had a tracked Salary for that month.
- **The Fix:** Now that dropdowns remain open after a deletion, deleting the final "Main" or "Self" item perfectly yields the empty table state message: *No expenses yet*. 
- **Month Retention:** Modified `groupByMonth` logic to prioritize the `salaries` database. A month card will now remain visible on the dashboard if it has a saved Salary, even if 0 expenses exist for that month. (If no salary and no expenses exist, the month correctly vanishes).

### 3. Refactored Expand/Collapse Controls
- Optimized the `expandAll()` and `collapseAll()` functions to batch-modify the new `openSectionIds` persistent Set and trigger a single re-render, ensuring cleaner and flawless interaction scaling.
